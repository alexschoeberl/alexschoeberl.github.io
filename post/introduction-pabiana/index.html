Introduction to Pabiana

Tutorial · Oct 04, 2017

Pabiana is a lightweight Python framework that enables you to build large cognitive applications distributed over several nodes. Its intended application area ranges across home automation systems, robot control systems and intelligent software assistants. The framework's networking capabilities are based on the ZeroMQ library. The functional principles are inspired by the human cortex. Pabiana therefore enforces a very specific style of application design and programming. This article will give you an overview of the framework and how to use it.

Installation

In the context of this article, the term cognitive
is used to describe systems that process sensory
input of multiple modalities, gain a thorough
understanding of the data and possibly control
actuators based on their insights.

Pabiana is a Python library. It is only tested against the CPython interpreter in its versions starting from 3.5, but might also work under different environments. The source code is published on Github. The framework requires recent versions of the PyZMQ and Pip Python packages, which will be downloaded during installation. Therefore, it is recommended to install Pabiana into a virtual environment.

To start, create a directory on your computer to contain your application. Inside it, initialize a virtual environment with a command like the following.

python3 -m venv .myvenv

The last parameter determines the name of the environment. After running the command, a new, possibly invisible directory should have been created inside your application folder. If you are using Git, make sure it will be ignored. By activating your virtual environment, you force Pabiana to be installed within this directory.

python3 -m pip install --upgrade pabiana

This command will install the package and all of its dependencies for your new project. Your system is now set up and ready to create and run a Pabiana application.

The Basics

A Pabiana application consists of various so-called Areas, which are a special type of Nodes. Each Node is contained in its own process, with the Nodes running on a single or multiple IP-connected computers. It must have an unique name and a possibly shared IP address. Besides, it occupies some selectable ports. Area and Node are classes found in the Pabiana package.

Clocks are a special type of Nodes. They are responsible for synchronizing the workflow of the different Areas. A Pabiana application therefore acts similar to a turn-based game. It needs at least one Clock to run properly.

You can now write a script for every type of area in which you create an instance of the Area class. A dictionary is required, to match the Node name with the corresponding IP address and ports. Before you start the node, you also have to specify the name of the Clock.

from pabiana import Area

			ifs = dict()
			ifs['area-name-pub'] = {'ip': '127.0.0.1', 'port': 15001}
			ifs['area-name-rcv'] = {'ip': '127.0.0.1', 'port': 15002}
			ifs['clock-name-pub'] = {'ip': '127.0.0.1', 'port': 15003}
			ifs['clock-name-rcv'] = {'ip': '127.0.0.1', 'port': 15004}

			area = Area(name='area-name', interfaces=ifs)
			area.subscribe(clock_name='clock-name')
			area.run()

As you can see, each Node occupies two ports, recognizable by the -pub and -rcv suffixes. One of them is used to publish data, while the other receives incoming requests. Other Areas can subscribe to the Publishing Interface. Your script is now ready to be run. Set the logging level of the default logger to 5 in order to see what is happening. It's important to note that you can start multiple Areas from a single one of your scripts. Just make sure to not hard-code their names, as they have to be unique. Use Ctrl-C to stop the process safely.

The Convenient Way

Usually, a Pabiana application consists of a large collection of different, complex Areas. To prevent your project from becoming messy, the framework provides some convenience functions. To use them, you have to stick to a specific project layout. Take a look at the demo project, to get a feeling.

  • area_type_1/
    • __init__.py
  • area_type_2/
    • __init__.py
    • interface.py
    • logic.py
    • requirements.txt
  • interfaces.json

As you can see, this time a package is used instead of a script for every type of area. Since the address and port information is shared across all Nodes, it is outsourced into a JSON file. Open the corresponding file in the demo project, to see how it should be structured. At the root of each package (in the init file), you have to provide an Area object named area. Also you have to create a Python dictionary named config, that specifies the objects setup values. The dictionary contains information on the Clock (mandatory) and the Publishing Interfaces the Area subscribes to. It's important to know that an Area can publish data on different channels, or Slots. Therefore, it is possible to subscribe to a single or multiple Slots of an Area. You can also choose the Slot of the Clock you want to listen to.

from pabiana import Area, repo

			area = Area(repo['area-name'], repo['interfaces'])
			config = {
			  'clock-name': 'clock',
			  'clock-slot': 'fast',
			  'subscriptions': {
			    'vision': ['edges', 'colors'],
			    'engine': None
			  }
			}

Utilizing this layout, you are provided with a default runner application and don't have to call the run method yourself. Pabiana's repo dictionary contains some global information, including the name of the current Area.

Messages from subscribed Areas are stored in a Deque, with the newest message at index 0. A message is always a dictionary and contains a time-rcvd key, that stores an integer representing the time the message was received. You can compare this value to the Areas time attribute.

recent = area.context['subscribed-area']['slot'][0]

The processing of data happens inside of functions you have to define. By using decorators, you can declare these functions as hooks to be called at specific events. The alteration hook is called whenever a message has arrived during the last round, while the pulse hook is called once for every tick of the Clock. There is also the register decorator, which registers a function as a Remote Trigger. Remote Triggers can be called by other Areas, to influence operation.

@area.register
			def start():
			  pass

			@area.register
			def stop():
			  pass

			@area.alteration
			def process():
			  pass

			@area.pulse
			def routine():
			  pass

These functions can be defined in the init file, but it is usually better to do it in other modules, like interface and logic. An Area can call Triggers of another Area with the objects trigger function. To make an Area call one of its own triggers, you should usually use its autoloop method. It's important to note that a trigger is never called more than once in a round.

area.trigger('target-name', 'trigger-name', {'parameter': 'value'})

Every Area type object provides a method to publish processed data to other Areas. The .publish(data[, slot]) method expects a dictionary and an optional slot name. For other Areas to receive these messages, they have to subscribe to them.

Running the Application

After you have created the first simple Areas, it's time to test their communication. At the beginning, you will most likely want to run them all on your local computer. You can start every process for every Area in its own terminal. However, you can also start multiple Areas at the same time, while Python runs the necessary processes in the background. 
python3 -m pabiana area_type_1:area1 area_type_1:area2 clock:clock1

Use the interpreter from the virtual environment to run the start-up command. You have to specify a variable number of parameters, each consisting of two parts, divided by a colon. The first part determines the template module (the type) for an Area requested to be run. The second part specifies its name, as it is found in interfaces.json.

Pabiana also provides you with a default Clock. Making use of it, you don't have to define your own. You still have to, however, declare its IP address and ports in your Interfaces file. The name of the default Clock is simply clock. Also, you can drop the Slot key from the configuration. To run it together with your Areas, append the -C option to the command.

Application Architecture

To use Pabiana successfully, you have to adapt a very specific architecture. A Pabiana application consists of a great number of Areas, arranged in a pyramid-shaped hierarchy, bound together by their communication channels. At the base of the pyramid you will usually find Areas which collect low-level sensory data or directly interact with the environment through speakers, motors and software APIs. Upper Areas combine sensory and status data from multiple lower Areas for processing. To influence those lower Areas, they make extensive use of their public Remote Triggers. Areas at the top see a rather abstract representation of the environment. The uppermost one has a vague control over everything. Information is published upwards, while commands are issued downwards.

Adapting this architecture allows you to build complex cognitive applications in a natural way, step by step. The application can be distributed over several computers with varying hardware. However, the more data is passed between the Nodes, the stronger the network connection has to be.
